# Основной TUI sshmon — дизайн

**Статус:** дизайн завершён, ожидает проверки пользователем
**Дата:** 2026-07-18
**Референсы:** Purple (`erickochen/purple`), socktop (`jasonwitty/socktop`)

## 1. Цель

Заменить глобальные вкладки sshmon на иерархический, keyboard-first интерфейс:

1. домашний экран показывает весь парк серверов и живую сводку выбранного сервера;
2. Enter открывает полноэкранный dashboard выбранного сервера;
3. процессы, порты, история, логи и Docker открываются отдельными экранами внутри сервера;
4. LLM-чат доступен глобальным оверлеем из любого экрана;
5. история метрик сохраняется в SQLite между запусками.

SSH-сбор остаётся agentless. На удалённые серверы ничего не устанавливается.

## 2. Выводы из референсов

### Purple

Используем:

- плотный список хостов как основной экран;
- keyboard-first навигацию, поиск, командную палитру и контекстную помощь;
- сводку выбранного хоста рядом со списком;
- явные экраны и оверлеи вместо глобальных вкладок;
- цвет и форму одновременно для отображения состояния.

Не переносим SSH-management функции Purple, не связанные с мониторингом.

### socktop

Используем:

- плотный dashboard одного сервера;
- компактные графики CPU, памяти, load и сети;
- отдельные панели дисков и процессов;
- разные частоты обновления для дешёвых и дорогих источников данных;
- небольшие компоненты рендера вместо одного монолитного TUI-файла.

Не переносим удалённый агент и WebSocket-транспорт socktop.

## 3. Информационная архитектура

Навигационный state machine содержит экраны:

```text
Fleet
└── ServerDashboard
    ├── Processes
    ├── Ports
    ├── History
    ├── Logs
    └── Containers
```

Оверлеи доступны поверх подходящих экранов:

- `ChatOverlay` — глобальный LLM-чат;
- `SearchOverlay` — поиск в текущем наборе данных;
- `CommandPalette` — переходы и доступные действия;
- `HelpOverlay` — контекстная помощь.

`Esc` сначала закрывает верхний оверлей, затем возвращает на предыдущий экран. Глобальных вкладок нет.

## 4. Экран Fleet

### Широкий терминал

```text
┌ sshmon ─ 12 серверов · 10 online · 2 проблемы ─────────────┐
│ СЕРВЕРЫ                         │ ВЫБРАННЫЙ СЕРВЕР           │
│ ● prod-api-1  CPU 34% MEM 61%  │ prod-api-1 · online        │
│ ▲ prod-db-1   CPU 92% MEM 78%  │ CPU 34%  ▁▂▃▅▃▂           │
│ ○ staging-2   offline           │ MEM 61%  ▃▄▅▅▆▅           │
│                                 │ Disk 72% · Load 1.2        │
│                                 │ 1 предупреждение           │
└ j/k выбрать · enter открыть · / поиск · c чат · ? помощь ──┘
```

- Левая область: имя, группа, online/stale/offline, CPU, память и число проблем.
- Правая область: текущие значения, короткая история и последние проблемы выбранного сервера.
- Серверы сохраняют порядок конфигурации; поиск и фильтры не меняют исходный порядок.
- Состояния обозначаются цветом и символом: online, warning, critical, stale, offline.

### Узкий терминал

При недостаточной ширине остаётся полноэкранный список. `v` открывает/закрывает сводку выбранного сервера. Интерфейс обязан сохранять управление при минимальном размере `60×16`; при меньшем размере показывается компактное сообщение с доступными `q`, `?` и resize.

### Клавиши Fleet

- `j/k`, стрелки, PgUp/PgDn — выбор;
- `Enter` — dashboard сервера;
- `/` — поиск;
- `g` — фильтр группы;
- `!` — только проблемные серверы;
- `v` — сводка в узком режиме;
- `c` — LLM-чат;
- `:` — командная палитра;
- `?` — помощь;
- `q` — выход.

## 5. Server Dashboard

```text
fleet / prod-api-1                              ● online · 14:32
┌ CPU + load ─────────┐ ┌ Memory ─────────────┐
│ 34% ▁▂▃▅▃▂  1.2     │ │ 61% ██████░░░░      │
├ Network ────────────┤ ├ Disks ──────────────┤
│ eth0 ↓12M ↑3M       │ │ / 72%  /data 48%    │
└─────────────────────┘ └──────────────────────┘
 p процессы · o порты · h история · l логи · d docker
 esc назад · c чат · ? помощь
```

Dashboard показывает:

- CPU и load 1/5/15 с короткой историей;
- память и swap;
- сеть по интерфейсам;
- диски, место и IO;
- uptime, hostname, время последнего успешного опроса;
- текущие проблемы;
- доступность процессов, Docker и источников журналов.

Клавиши: `p` процессы, `o` порты, `h` история, `l` логи, `d` Docker, `r` обновить, `Esc` назад, `c` чат.

## 6. Глубокие экраны

### Processes

Read-only сортируемая таблица `PID`, `CPU%`, `MEM%/RSS`, `USER`, `COMMAND`, uptime процесса. Сортировка по CPU, памяти и PID; `/` фильтрует. Enter открывает read-only карточку с полной командой и связанными портами. Kill, renice и сигналы не входят в первую версию.

### Ports

Таблица `PROTO`, `LOCAL`, `STATE`, `PID`, `PROCESS`. Фильтр работает по порту, адресу и процессу. Внешние listening-адреса визуально отличаются от loopback. Enter переходит к связанному процессу, если PID известен.

### Containers

Docker только для наблюдения:

- имя, image, state, uptime, CPU, RAM, рестарты;
- Enter открывает детали и live-логи контейнера;
- start/stop/restart/exec отсутствуют;
- отсутствие Docker или прав отображается как `unsupported`, не переводит сервер в offline.

### Logs

Источники: systemd unit, файл или Docker-контейнер. Экран поддерживает live tail, pause/resume, текстовый фильтр и перенос строк. В памяти хранится не более 10 000 последних строк. Разрыв SSH оставляет уже полученные строки и предлагает `r` для переподключения. Выход с экрана отменяет удалённую команду.

### History

Диапазоны: `1h`, `6h`, `24h`, `7d`, `30d`. Графики: CPU, память, load, network RX/TX и максимально заполненный диск. Курсор показывает точное время и значения. Короткие диапазоны читают raw-замеры, длинные — минутные агрегаты.

## 7. Глобальный LLM-чат

`c` открывает выдвижной оверлей из любого экрана. System context содержит:

- текущий снимок всего парка;
- активный сервер;
- активный экран и видимые данные;
- последние проблемы;
- запрошенный фрагмент истории, если открыт History.

Чат не блокирует сбор метрик и закрывается по `Esc`. История самого чата остаётся сессионной; хранение диалогов в SQLite не входит в этот релиз.

## 8. SQLite-история

Используем `database/sql` и pure-Go драйвер `modernc.org/sqlite`, без CGO. Путь по умолчанию:

```text
~/.local/share/sshmon/history.db
```

Конфигурация:

```yaml
history:
  enabled: true
  raw_retention: 24h
  aggregate_retention: 720h
```

### Идентичность сервера

`server_key` детерминированно строится из connection identity: пользователь сохраняется без изменения регистра, hostname приводится к нижнему регистру, порт сначала нормализуется (`0` → `22`), а адрес собирается через эквивалент `net.JoinHostPort` для однозначной поддержки IPv6. Итоговый формат — `user@host:port`, где IPv6 заключён в квадратные скобки. Display name хранится отдельно. Изменение пользователя, адреса или порта начинает новую историю; переименование display name без изменения connection identity не теряет историю.

### Таблицы

`metric_samples`:

- `server_key`, `sampled_at_ms` — составной primary key;
- online-флаг;
- CPU, memory, swap, load1/5/15;
- суммарные RX/TX bytes per second;
- максимальная заполненность диска;
- количество warning/critical issues.

Для offline-замера записываются `online=false`, время и число issues; поля метрик остаются `NULL`, чтобы недоступность сервера не превращалась в нулевую загрузку. Offline-строки не участвуют в min/max/avg rollup, но используются для отображения разрывов на графике.

`metric_rollups_minute`:

- `server_key`, `bucket_at_ms` — составной primary key;
- count;
- min/max/avg для CPU, memory, load, RX/TX и disk usage.

### Политика хранения

- raw-замеры: 24 часа;
- минутные rollup: 30 дней;
- сроки задаются конфигом;
- агрегация и удаление выполняются небольшими транзакциями;
- один writer последовательно принимает результаты опросов;
- UI читает историю отдельными запросами;
- DSN включает WAL, `synchronous=NORMAL`, foreign keys и `busy_timeout`.

Если SQLite недоступен, текущий мониторинг продолжает работать. UI показывает `history unavailable`; запись автоматически повторяется после следующего опроса без бесконечного retry-loop.

## 9. Сбор данных и частоты

- основные `/proc`-метрики — по существующему `interval`;
- запись history — после каждого успешного или offline-опроса;
- процессы — каждые 2 секунды, только пока открыт Processes;
- порты — при открытии и каждые 5 секунд на экране Ports;
- Docker — при открытии и каждые 5 секунд на экране Containers;
- графики — чтение SQLite при входе, смене диапазона и поступлении нового bucket;
- logs — отдельная отменяемая SSH-сессия с bounded channel.

Уход с глубокого экрана отменяет его context. Результат от старого экрана содержит screen/request ID и игнорируется после смены экрана.

## 10. Состояния и ошибки

Каждая панель имеет состояния `loading`, `ready`, `stale`, `unsupported`, `error`.

- Ошибка Processes, Docker, Logs или SQLite не делает сервер offline.
- Offline определяется основным SSH-опросом.
- Последние успешные данные остаются видимыми с отметкой возраста.
- Пустой набор отображает объяснение, а не пустую рамку.
- Ошибки пользователя показываются в нижней status line и не заменяют весь экран.
- Все удалённые команды имеют timeout или cancellation.
- Docker и процессы в первой версии строго read-only.

## 11. Модульные границы

Текущий `internal/tui/tui.go` не расширяется дальше. Реализация должна разделить ответственность на небольшие файлы/модели:

- корневая модель приложения и навигационный state machine;
- Fleet и preview;
- Dashboard и отдельные панели;
- Processes, Ports, History, Logs, Containers;
- overlays: chat, search, command palette, help;
- adaptive layout и общая status/help line.

Новый `internal/history` владеет SQLite schema, writer, query и retention. Новые SSH collectors для процессов, Docker и stream logs остаются в `internal/collect`/`internal/sshx`, но не смешиваются с рендером.

Production-файлы должны оставаться фокусными; целевой предел — до 250 чистых строк на файл.

## 12. Совместимость и non-goals

Сохраняются:

- Bubble Tea v1, Bubbles v1 и Lip Gloss v1;
- agentless SSH;
- существующий YAML серверов и thresholds;
- `--headless` и текущий MCP-контракт;
- LLM providers;
- first-run SSH picker.

Не входят:

- установка удалённого агента;
- управление Docker;
- kill/renice процессов;
- ProxyJump/ProxyCommand;
- хранение истории LLM-диалогов;
- миграция Bubble Tea v2;
- обязательное расширение MCP новыми history/process/container tools в этом релизе.

## 13. Проверка

### Unit

- навигационный state machine и стек оверлеев;
- адаптивная раскладка и минимальные размеры;
- фильтры/сортировка серверов, процессов и портов;
- графические downsampling helpers;
- bounded log buffer;
- SQLite rollup и retention;
- server key.

### Integration

- collector → history writer → history query;
- миграции новой и существующей БД;
- отмена процессов, Docker polling и live tail;
- stale result после смены экрана игнорируется;
- SQLite unavailable не останавливает Collector.

### PTY E2E

Размеры: `80×24`, `120×30`, `160×40`.

Сценарии:

1. Fleet, поиск, фильтр проблем и drill-down;
2. dashboard и возврат без потери выбранного сервера;
3. смена диапазона History и cursor inspection;
4. переход Port → Process;
5. Docker unavailable и Docker read-only details;
6. live log, pause, disconnect, reconnect и cleanup;
7. глобальный чат с корректным active-server context;
8. resize каждого экрана и сохранение управления.

Финальные гейты: gofmt, build, vet, race+shuffle tests, binary build, diff check и ручной PTY smoke.

## 14. Критерии готовности

- Глобальных вкладок нет.
- Fleet остаётся основным экраном и показывает preview выбранного сервера.
- Все глубокие экраны доступны клавиатурой и возвращаются по `Esc`.
- Dashboard остаётся читаемым на `80×24` и использует дополнительное пространство на широком терминале.
- История переживает перезапуск и соблюдает retention 24h/30d.
- Processes, Ports, Logs и Docker не запускают постоянную нагрузку, когда экран закрыт.
- Docker/process функции read-only.
- Ошибка одного источника не ломает основной мониторинг.
- PTY E2E и полные Go-гейты проходят.
